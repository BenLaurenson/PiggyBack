# Subdomain System (Phase 3.2)

## Audit summary (current behaviour)

Every hosted-platform tenant gets `<shortid>.piggyback.finance` assigned at the moment they finish Google sign-in. The shortid is a 6-character base32 string generated by `generateShortId()` in `src/lib/provisioner/subdomain.ts` (Crockford-ish alphabet — no `i/l/o/u/0/1`), checked for collisions against the orchestrator's `piggyback_provisions.subdomain_short_id` column with up to five retries inside `upsertProvisionForUser` (`src/lib/provisioner/state-machine.ts`), and persisted as the user's row is created. The provisioner state machine then walks SUPABASE_AUTHED → SUPABASE_PROVISIONED → MIGRATIONS_RUN → VERCEL_PROVISIONED → ENV_VARS_SET → DOMAIN_ATTACHED, and at the DOMAIN_ATTACHED step `attachSubdomain` in `src/app/api/provision/execute/route.ts` calls `addProjectDomain` on the user's Vercel project (using their stored OAuth token from `provision_oauth_tokens`) with `buildHostname(subdomain_vanity ?? subdomain_short_id)` — so the same code path serves both the default shortid and any vanity rename. Vanity renames are exposed via `POST /api/account/subdomain` (Phase 3.2 addition): the endpoint validates against the regex `^[a-z0-9]([a-z0-9-]{1,30}[a-z0-9])?$`, the `RESERVED_VANITY_NAMES` set (which covers admin/api/app/www/auth/oauth/account/billing/dashboard/login/signup/admin/buck/penny/piggyback/openclaw/etc. plus all 1- and 2-char names), enforces a 30-day rate limit via `subdomain_vanity_set_at` and `vanityChangeAllowedFrom`, and on success attaches the new hostname to the user's Vercel project, detaches the old hostname from the user's project, re-attaches the old hostname to the orchestrator's Vercel project (so middleware sees it), records a row in `subdomain_aliases` with a 30-day `expires_at`, and updates `piggyback_provisions.subdomain_vanity`. The orchestrator `src/middleware.ts` checks every incoming request against `subdomain_aliases` via `resolveSubdomainAliasRedirect` and 301s any active alias to the tenant's current hostname; an hourly cron at `/api/admin/release-aliases` sweeps expired alias rows, removes the corresponding domain from the user's Vercel project (best-effort), and deletes the alias row.

## Environment variables

- `ORCHESTRATOR_VERCEL_TOKEN` (or `VERCEL_API_TOKEN` fallback) — system token used to re-attach retired hostnames to the orchestrator project so middleware can serve the 301.
- `ORCHESTRATOR_VERCEL_PROJECT_ID` — target project for retired-hostname attaches.
- `ORCHESTRATOR_VERCEL_TEAM_ID` — optional team scope for the orchestrator token.
- `NEXT_PUBLIC_HOSTED_ENABLED` — middleware skips alias lookups unless this is `"true"` (so tenant deploys / self-hosters don't try to read a table they don't own).
- `CRON_SECRET` — gates `/api/admin/release-aliases` Bearer auth.
